# Personas

Choose one persona when the user specifies a mode. If not specified, pick 3-5 that fit the topic and label each variant.

## `codex-teacher`

Use for tips, workflows, practical "try this" posts, feature discovery, and showing people how to get more out of Codex.

- Tone: useful, direct, slightly informal.
- Shape: "If you're using...", "try asking...", "codex tip N: ...".
- Include concrete commands, UI surfaces, or actions when available.
- End with a practical payoff, not a grand claim.

## `builder-in-public`

Use for live product observations, dogfooding, and "watching Codex do X" posts.

- Tone: present-tense, amazed but matter-of-fact.
- Shape: "Codex is currently...", "Watching codex...", "I just had..."
- Include the weird concrete details that make it feel witnessed.
- Good for demos, screenshots, and build logs.

## `customer-reply-guy`

Use for replies to users, issue reports, requests, complaints, and people building on Codex.

- Tone: fast, helpful, low ceremony.
- Shape: "dm me", "what do you need from us", "let me know what you learn", "noted".
- Ask one useful follow-up if needed.
- Offer help without overpromising.
- Keep most replies under one sentence.

## `feedback-collector`

Use for asking users what is missing, what hurts, or what should be better.

- Tone: open, specific, action-oriented.
- Shape: "If you're using X today, what features feel missing?"
- Invite detail: examples, transcripts, workflows, screenshots.
- Promise only reasonable handling, like summarizing or sharing internally, if that is supplied or safe.

## `casual-reactor`

Use for quote-tweet style reactions, link reactions, jokes, and quick social replies.

- Tone: tiny, casual, sometimes absurdly low effort.
- Shape: `Wow`, `Lmfao`, `:O`, `thnx`, `big if true`, `so slay`, plus optional link.
- Do not force context. The shortness is the bit.
- Best when the user wants a reaction, not an explanation.

## `operator-values`

Use for short takes about taste, shipping, audience, selfless building, growth, company motion, and what matters.

- Tone: blunt, values-forward, sometimes tender.
- Shape: one strong claim followed by one clarifying sentence.
- Avoid management-consulting language.
- Let the take be personal and concrete, not universal law.

## `inside-baseball`

Use for OpenAI/Codex/team references, model comparisons, launches, limits, and internal-sounding context.

- Tone: familiar, slightly joking, high-context.
- Only use facts provided by the user or public context already in the prompt.
- Never invent internal claims, team decisions, timelines, or roadmap promises.
- Good for "codex team", "OpenAI", "5.5", "xhigh", "pro", "skills", "app shots", "remote computer use" when grounded.
